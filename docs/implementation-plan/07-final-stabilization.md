# Этап 07: Final Stabilization

## Цель

Привести проект к финальному стабильному состоянию MVP: убрать временный код, проверить соответствие RFC и убедиться, что приложение запускается одной командой.

## Начало этапа

Начинать этап можно, если завершен этап 06:

- Пользователь может отправить сообщение из frontend.
- Ответ ассистента приходит через WebSocket stream.
- История сохраняется в MongoDB.
- Markdown отображается безопасно.

## Задачи

### 1. Удалить временный код

Удалить:

- Weather endpoints, если они больше не нужны.
- Демо UI.
- Временные debug endpoints, если они были добавлены.
- Неиспользуемые типы, стили и переменные.

Не удалять полезные health/root endpoints, если они помогают проверить запуск.

### 2. Проверить соответствие RFC

Сверить реализацию с `docs/rfc-ai-agent-chat.md`.

Обязательно проверить:

- MongoDB является источником данных для чатов и истории.
- Local storage и IndexedDB не используются для хранения истории.
- Используется модель `qwen3.5:4b`.
- Ollama вызывается через `/api/chat`.
- Backend вызывает Ollama через Refit.
- Настройки читаются через `IOptions<>`.
- WebSocket endpoint: `/ws/chats/{chatId}`.
- `chatId` не передается в WebSocket payload.
- Максимальный размер сообщения: 4000 символов.
- Максимум чатов: 10 для неявного пользователя MVP.
- Markdown рендерится безопасно.
- UI использует черно-белую цветовую схему.

### 3. Проверить Docker Compose

Убедиться, что compose содержит:

- `backend`
- `frontend`
- `mongo`
- `ollama`
- volume `mongo_data`
- volume для Ollama data

Ollama-модель считается загруженной заранее. Не добавлять автоматический pull модели.

### 4. Ручная проверка MVP

Проверить сценарии вручную:

- Открыть приложение.
- Создать чат.
- Отправить сообщение до 4000 символов.
- Получить streaming-ответ.
- Увидеть thinking, если модель его вернула.
- Не увидеть ошибку, если thinking не вернулся.
- Перезагрузить страницу.
- Убедиться, что чат и история восстановились.
- Создать 10 чатов.
- Убедиться, что 11-й чат создать нельзя.
- Переименовать чат.
- Удалить чат.

### 5. Обновить документацию при необходимости

Если в реализации появились отличия от RFC, не скрывать их.

Варианты:

- Исправить реализацию под RFC.
- Или обновить RFC, если изменение осознанное и согласовано.

## Конец этапа

Этап завершен, когда:

- Приложение запускается одной командой.
- Frontend доступен на `http://localhost:3000`.
- Backend доступен на `http://localhost:5000`.
- MongoDB хранит чаты и историю.
- Ollama используется через backend.
- WebSocket-чат работает.
- Временный код удален.
- Документация соответствует реализации.

## Проверки

Выполнить:

```powershell
dotnet build AI-Agent-vaity.sln
npm run build --prefix frontend
docker compose config
docker compose up --build
```

Проверить доступность:

```powershell
Invoke-WebRequest -UseBasicParsing http://localhost:3000
Invoke-WebRequest -UseBasicParsing http://localhost:5000
```

Остановить стек:

```powershell
docker compose down
```

## Стабильное состояние после этапа

Проект готов как MVP согласно RFC. Его можно запускать командой:

