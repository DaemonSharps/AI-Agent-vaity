# Этап 06: Frontend WebSocket и Markdown

## Цель

Подключить frontend к WebSocket-чату, реализовать streaming-ответы, отображение thinking и безопасный рендер базового Markdown.

## Начало этапа

Начинать этап можно, если завершен этап 05:

- Frontend отображает shell чата.
- HTTP API управления чатами подключен.
- Backend WebSocket endpoint `/ws/chats/{chatId}` работает.

## Задачи

### 1. Подключить WebSocket

Frontend должен открывать соединение:

```text
ws://localhost:5000/ws/chats/{chatId}
```

Для production URL строить от `VITE_API_URL`, заменяя `http` на `ws` и `https` на `wss`.

При переключении чата старое WebSocket-соединение нужно закрывать.

### 2. Отправлять пользовательское сообщение

Payload должен быть только таким:

```json
{
  "type": "user_message",
  "content": "Привет"
}
```

Не добавлять `chatId` в payload.

Перед отправкой проверить:

- Сообщение не пустое.
- Длина сообщения не больше 4000 символов.
- Нет уже активной отправки, если повторная отправка может создать дубль.

### 3. Обрабатывать WebSocket-события

Обработать события:

- `user_message_saved`
- `assistant_started`
- `assistant_thinking_delta`
- `assistant_delta`
- `assistant_complete`
- `error`

Правила:

- `assistant_started` создает или показывает assistant-сообщение со статусом `streaming`.
- `assistant_delta` дописывает текст в `content`.
- `assistant_thinking_delta` дописывает текст в `thinking`.
- Если thinking-событий нет, UI не должен показывать ошибку.
- `assistant_complete` переводит сообщение в `complete`.
- `error` показывает понятную ошибку и не очищает введенный пользователем текст без необходимости.

### 4. Отображать состояние генерации

UI должен показывать:

- Что сообщение отправляется.
- Что ассистент генерирует ответ.
- Ошибку при сбое WebSocket или API.

### 5. Реализовать безопасный Markdown

Сообщения ассистента должны рендериться как базовый Markdown.

Поддержать:

- Абзацы.
- Заголовки.
- Списки.
- Ссылки.
- Цитаты.
- Inline-код.

Не включать:

- Таблицы.
- Блоки кода.
- Raw HTML.

Требования безопасности:

- Raw HTML должен быть отключен или санитизирован.
- Ссылки должны санитизироваться.
- Внешние ссылки должны открываться в новой вкладке.
- Не использовать небезопасную вставку HTML без санитизации.

Допустимый вариант реализации: `react-markdown` без `rehype-raw`, с настройкой ссылок.

### 6. Обновлять историю после ответа

После `assistant_complete` можно:

- Оставить уже собранное сообщение в состоянии frontend.
- Или повторно загрузить историю сообщений через `GET /api/chats/{chatId}/messages`.

Для MVP предпочтительно повторно загрузить историю после завершения, чтобы синхронизироваться с MongoDB.

## Конец этапа

Этап завершен, когда:

- Пользователь может отправить сообщение из UI.
- Сообщение уходит через WebSocket.
- Ответ ассистента отображается потоково.
- Thinking отображается отдельно, если приходит.
- Отсутствие thinking не ломает UI.
- После ответа история сохраняется в MongoDB и восстанавливается после перезагрузки.
- Markdown отображается безопасно.
- Проект собирается.

## Проверки

Выполнить:

```powershell
npm run build --prefix frontend
dotnet build AI-Agent-vaity.sln
docker compose config
docker compose up --build
```

Ручная проверка:

- Открыть `http://localhost:3000`.
- Создать чат.
- Отправить сообщение.
- Убедиться, что ответ появляется частями.
- Перезагрузить страницу.
- Убедиться, что история чата восстановилась из MongoDB.

Остановить стек:

```powershell
docker compose down
```

## Стабильное состояние после этапа

MVP-чат работает end-to-end через frontend, WebSocket, backend, MongoDB и Ollama.
