# RFC: чат с AI-агентом

## Статус

Черновик

## Краткое описание

Документ описывает требования к базовому интерфейсу чата с AI-агентом. Пользователь может отправлять сообщения, получать ответы ассистента, видеть историю диалога, сохранять контекст и переключаться между несколькими чатами.

## Цели

- Предоставить простой интерфейс для общения с AI-агентом.
- Сохранять историю и контекст диалога.
- Позволить пользователю переключаться между чатами.
- Ограничить количество чатов до 10 штук.
- Поддержать базовый Markdown в ответах ассистента.
- Обеспечить понятные состояния загрузки и ошибки.
- Использовать WebSocket для обмена сообщениями чата.
- Использовать streaming-ответы и think-режим модели.
- Хранить все чаты и историю сообщений в MongoDB.
- Использовать Ollama API как AI-провайдера.
- Безопасно отображать Markdown-ответы ассистента.

## Не входит в цели

- Закрепление, архивация и поиск чатов.
- Вложения файлов.
- Экспорт и импорт чатов.
- Настройка модели, temperature, лимитов токенов и системного промпта через UI.
- Оценка ответов и сбор обратной связи.
- Отображение токенов, стоимости или лимитов использования.
- Поддержка совместной работы нескольких пользователей в реальном времени.
- Реализация административной панели.
- Реализация биллинга или платежных сценариев.

## Принятые технические решения

- Хранилище данных: MongoDB.
- История чатов хранится на backend в MongoDB.
- Frontend получает список чатов и историю сообщений с backend, а не из local storage или IndexedDB.
- AI-провайдер: Ollama API.
- Ollama запускается как отдельный сервис в `docker-compose.yml`.
- Для обращения к Ollama используется endpoint `/api/chat`.
- Модель Ollama: `qwen3.5:4b`.
- Модель `qwen3.5:4b` считается предварительно загруженной, приложение не должно выполнять `ollama pull`.
- Backend вызывает Ollama API через Refit.
- Настройки Ollama должны быть вынесены в `appsettings.json`.
- Доступ к настройкам Ollama на backend должен выполняться через `IOptions<>`.
- Транспорт для чата: WebSocket.
- Ответы модели должны передаваться во frontend потоково.
- Для модели должен использоваться think-режим, если он поддерживается выбранной моделью Ollama.
- Если модель не возвращает think-содержимое, frontend должен показывать только основной ответ без ошибки.
- Максимальный размер пользовательского сообщения: 4000 символов.
- Основные цвета интерфейса: белый и черный.

## Требования к пользовательскому опыту

### Ввод сообщения

- Интерфейс должен предоставлять текстовое поле для ввода сообщений.
- Пользователь должен иметь возможность отправить сообщение кнопкой отправки.
- Пользователь должен иметь возможность отправить сообщение клавишей `Enter`.
- Пользователь должен иметь возможность вставить новую строку через `Shift + Enter`.
- Пустые сообщения не должны отправляться.
- Поле ввода должно показывать сообщение валидации, если текст превышает 4000 символов.
- Приложение должно предотвращать дублирующую отправку из-за повторных кликов или нажатий клавиш.

### История сообщений

- Интерфейс должен отображать историю текущего чата.
- Сообщения пользователя и ассистента должны визуально отличаться.
- Каждое сообщение должно отображать время создания.
- Сообщения должны отображаться в хронологическом порядке.
- Область сообщений должна автоматически прокручиваться к последнему сообщению, когда это уместно.
- Если пользователь прокрутил историю вверх, приложение не должно принудительно возвращать его вниз.
- Если пользователь находится не внизу истории, приложение должно показывать кнопку перехода к последнему сообщению.

### Состояние ответа ассистента

- Приложение должно показывать, что ассистент генерирует ответ.
- Приложение должно отображать ответ ассистента потоково по мере получения частей ответа через WebSocket.
- Приложение должно отображать think-часть ответа отдельно от основного ответа, если модель возвращает такую информацию.
- Пользователь должен иметь возможность повторить неудачный запрос.
- Приложение должно показывать понятное сообщение об ошибке, если ответ получить не удалось.

## Требования к управлению чатами

### Несколько чатов

- Пользователь должен иметь возможность создать новый чат.
- Пользователь должен иметь возможность переключаться между существующими чатами.
- Приложение должно сохранять независимую историю сообщений для каждого чата.
- Приложение должно сохранять контекст каждого чата.
- Текущий чат должен иметь видимый заголовок.
- Приложение должно автоматически формировать заголовок по первым сообщениям.
- Пользователь должен иметь возможность переименовать чат.
- Пользователь должен иметь возможность удалить чат.
- Приложение должно запрашивать подтверждение перед удалением чата.
- Максимальное количество чатов: 10.
- Максимальное количество чатов считается на пользователя.
- В MVP авторизация не реализуется, поэтому используется один неявный пользователь приложения.
- Если неявный пользователь уже создал 10 чатов, кнопка создания нового чата должна быть недоступна или должна показывать понятное сообщение об ограничении.

### Список чатов

- Интерфейс должен отображать список чатов.
- Список должен поддерживать пустое состояние.
- Список должен показывать не более 10 чатов.
- Каждый элемент списка должен отображать название чата.
- Активный чат должен быть визуально выделен.

## Требования к Markdown

- Сообщения ассистента должны поддерживать базовый Markdown.
- Markdown должен поддерживать абзацы, заголовки, списки, ссылки, цитаты и inline-код.
- Ссылки должны быть кликабельными.
- Внешние ссылки должны открываться в новой вкладке браузера.
- Markdown должен рендериться безопасно.
- Raw HTML внутри Markdown должен быть отключен или санитизирован.
- Ссылки должны санитизироваться перед отображением.
- Сгенерированный Markdown не должен вставляться в DOM через небезопасные API без санитизации.

## Требования к контексту и хранению

### Контекст диалога

- Приложение должно сохранять контекст каждого чата.
- Контекст должен сохраняться после перезагрузки страницы.
- Контекст должен восстанавливаться при возвращении к чату.
- Приложение должно поддерживать роли сообщений: `user` и `assistant`.
- Для MVP backend должен сохранять и использовать контекст, который возвращает Ollama.
- Управление контекстом выполняет Ollama, backend не должен самостоятельно пересчитывать или сокращать контекст в первой версии.
- История сообщений хранится в MongoDB для отображения пользователю.

### Хранение данных

- Каждый чат должен иметь стабильный идентификатор.
- Каждое сообщение должно иметь стабильный идентификатор.
- Каждое сообщение должно хранить роль, содержимое, время создания, порядок и статус.
- Каждое сообщение ассистента должно иметь возможность хранить think-содержимое отдельно от основного ответа.
- Статус сообщения должен поддерживать как минимум значения: `pending`, `sent`, `streaming`, `failed`, `complete`.
- MongoDB является единственным источником данных для чатов и истории сообщений.
- Local storage и IndexedDB не используются для хранения истории чатов в первой версии.
- Контекст Ollama должен храниться в MongoDB вместе с чатом или отдельной связанной записью.

### Настройки MongoDB

- Строка подключения MongoDB должна храниться в `appsettings.json`.
- Имя базы данных должно храниться в `appsettings.json`.
- Доступ к настройкам MongoDB на backend должен выполняться через `IOptions<>`.
- В docker compose должен быть отдельный сервис MongoDB.
- Backend должен подключаться к MongoDB по имени сервиса внутри docker network.
- Для MongoDB должен быть настроен named volume в `docker-compose.yml`, чтобы история чатов и контекст не терялись при пересоздании контейнеров.

## API-контракт

- Frontend должен использовать HTTP API для управления чатами и загрузки истории.
- Frontend должен использовать WebSocket для отправки сообщений и получения потоковых ответов ассистента.
- Идентификатор чата должен передаваться только в URL WebSocket endpoint.
- WebSocket-сообщение от frontend не должно дублировать `chatId` в payload.
- Backend должен загружать сохраненный Ollama context текущего чата из MongoDB и передавать его в Ollama.
- Ответ ассистента должен передаваться через WebSocket частями.
- Think-содержимое должно передаваться отдельными WebSocket-событиями, если оно есть в ответе модели.
- Если think-содержимое отсутствует, backend не должен отправлять `assistant_thinking_delta`.
- Ошибки должны передаваться через WebSocket в едином формате.
- Frontend должен обрабатывать обрыв WebSocket-соединения, timeout и повторную попытку после ошибки.

## Требования к backend

- Backend должен быть реализован на .NET 9 Minimal API.
- Backend должен использовать MongoDB для хранения чатов и сообщений.
- Backend должен использовать Refit для вызова Ollama API.
- Backend должен использовать `IOptions<>` для доступа к настройкам MongoDB и Ollama.
- Backend должен предоставлять HTTP endpoints для управления чатами.
- Backend должен предоставлять WebSocket endpoint для обмена сообщениями чата.
- Backend должен сохранять пользовательское сообщение в MongoDB перед вызовом Ollama.
- Backend должен сохранять ответ ассистента в MongoDB после завершения генерации.
- Backend должен обновлять статус сообщения ассистента во время генерации: `streaming`, затем `complete` или `failed`.
- Backend должен передавать во frontend части ответа Ollama по мере получения stream-событий.
- Backend должен передавать think-содержимое отдельно от основного текста ответа, если Ollama возвращает такую информацию.
- Backend должен валидировать максимальную длину пользовательского сообщения: 4000 символов.
- Backend должен сохранять обновленный Ollama context после завершения ответа модели.

## Настройки приложения

### appsettings.json

```json
{
  "MongoDb": {
    "ConnectionString": "mongodb://mongo:27017",
    "DatabaseName": "ai_agent_chat"
  },
  "Ollama": {
    "BaseUrl": "http://ollama:11434",
    "Model": "qwen3.5:4b",
    "UseStream": true,
    "UseThink": true
  }
}
```

### Options-классы

```csharp
public sealed class MongoDbOptions
{
    public string ConnectionString { get; init; } = string.Empty;
    public string DatabaseName { get; init; } = string.Empty;
}

public sealed class OllamaOptions
{
    public string BaseUrl { get; init; } = string.Empty;
    public string Model { get; init; } = string.Empty;
    public bool UseStream { get; init; }
    public bool UseThink { get; init; }
}
```

## Предлагаемая модель данных

### Chat

```ts
type Chat = {
  id: string;
  title: string;
  createdAt: string;
  updatedAt: string;
  ollamaContext?: number[];
};
```

### Message

```ts
type Message = {
  id: string;
  chatId: string;
  role: 'user' | 'assistant';
  content: string;
  thinking?: string;
  createdAt: string;
  status: 'pending' | 'sent' | 'streaming' | 'failed' | 'complete';
  order: number;
};
```

## Docker Compose

- В `docker-compose.yml` должны быть сервисы backend, frontend, MongoDB и Ollama.
- MongoDB должен использовать named volume `mongo_data`.
- Ollama считается подготовленным заранее: модель `qwen3.5:4b` уже загружена.

```yaml
services:
  mongo:
    image: mongo:7
    volumes:
      - mongo_data:/data/db

volumes:
  mongo_data:
```

## Предлагаемые API endpoints

### WebSocket чат

```text
GET /ws/chats/{chatId}
```

Сообщение от frontend:

```json
{
  "type": "user_message",
  "content": "Привет"
}
```

Событие создания сообщения пользователя:

```json
{
  "type": "user_message_saved",
  "messageId": "msg_user_123",
  "status": "sent"
}
```

Событие начала ответа ассистента:

```json
{
  "type": "assistant_started",
  "messageId": "msg_assistant_123",
  "status": "streaming"
}
```

Потоковое сообщение от backend:

```json
{
  "type": "assistant_delta",
  "messageId": "msg_assistant_123",
  "content": "Здравствуйте"
}
```

Think-сообщение от backend:

```json
{
  "type": "assistant_thinking_delta",
  "messageId": "msg_assistant_123",
  "content": "Пользователь поздоровался, нужно ответить приветствием."
}
```

Backend отправляет `assistant_thinking_delta` только если модель вернула think-содержимое.

Событие завершения:

```json
{
  "type": "assistant_complete",
  "messageId": "msg_assistant_123",
  "status": "complete"
}
```

Событие ошибки:

```json
{
  "type": "error",
  "message": "Не удалось получить ответ от модели."
}
```

Идентификатор чата не передается в payload WebSocket-сообщений, потому что он уже указан в URL `/ws/chats/{chatId}`.

### Управление чатами

```http
GET /api/chats
POST /api/chats
GET /api/chats/{chatId}
PATCH /api/chats/{chatId}
DELETE /api/chats/{chatId}
```

### Получение истории чата

```http
GET /api/chats/{chatId}/messages
```

## Интеграция с Ollama

- Backend должен обращаться к Ollama API через Refit-клиент.
- Backend должен использовать Ollama endpoint `/api/chat`.
- Base URL Ollama должен задаваться через `Ollama:BaseUrl` в `appsettings.json`.
- Название модели должно задаваться через `Ollama:Model` в `appsettings.json`.
- Значение `Ollama:Model` для MVP: `qwen3.5:4b`.
- Модель считается предварительно загруженной в Ollama.
- Параметр stream должен быть включен через `Ollama:UseStream`.
- Think-режим должен быть включен через `Ollama:UseThink`, если выбранная модель его поддерживает.
- Если think-режим не возвращает отдельное содержимое, это не считается ошибкой.
- Backend должен передавать сохраненный Ollama context текущего чата в `/api/chat`.
- Backend должен сохранять новый Ollama context из ответа `/api/chat` после завершения генерации.
- Backend должен преобразовывать stream-ответ Ollama в WebSocket-события для frontend.

## Требования к надежности

- Ошибки API должны показываться как понятные сообщения для пользователя.
- Сетевые ошибки должны обрабатываться корректно.
- Приложение не должно терять уже введенный пользователем текст после ошибки отправки.
- Приложение должно показывать состояние загрузки во время отправки сообщения.
- Приложение должно показывать состояние генерации ответа ассистента.

## Требования к интерфейсу

- Интерфейс должен работать на desktop-экранах.
- Интерфейс должен работать на мобильных экранах.
- Список чатов должен сворачиваться или перемещаться в отдельную панель на маленьких экранах.
- Для нового чата и пустого списка чатов должны быть предусмотрены пустые состояния.
- Основная цветовая схема интерфейса должна использовать белый и черный цвета.
- Дополнительные цвета допускаются только для ошибок, disabled-состояний и минимальных UI-акцентов.
